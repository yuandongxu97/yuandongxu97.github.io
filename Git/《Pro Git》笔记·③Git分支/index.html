《Pro Git》笔记·③Git分支

发表于 2020-03-04 本文字数： 1.5k 标签： Git Valine：

几乎每一种版本控制系统都以某种形式支持分支。使用分支意味着你可以从开发主线上分离开来，然后在不影响主线的同时继续工作。

有人把 Git 的分支模型称为必杀技特性，而正是因为它，将 Git 从版本控制系统家族里区分出来。

分支使用

HEAD为当前分支指针

branch 分支创建

创建分支实际上就是创建一个新的可移动指针，比如说创建testing分支：git branch testing

checkout 分支切换

比如从当前的master切换到testing分支：git checkout testing

或者直接创建并切换到一个不存在的分支：git checkout -b newbranch

然后再提交一次，这样当前HEAD也就是testing分支前进但master停留在原地。

分支切换会改变工作目录

git checkout master，使用命令的后果：

  1. 使 HEAD 指回 master 分支
  2. 将工作目录恢复成 master 分支所指向的快照内容。 也就是说,你现在做修改的话,项目将始于一个较旧的版本。 本质上来讲,这就是忽略 testing 分支所做的修改,以便于向另一个方向进行开发。

此时再提交一次，master前进，此时会形成分支的分叉

分支合并

分支的合并有两种情况：

  • 无分叉，直接前进
    当你试图合并两个分支时, 如果顺着一个分支走下去能够到达另一个分支,那么 Git 在合并两者的时候, 只会简单的将指针向前推进(指针右移)，因为该情况下没有产生分歧文件，也被称为快进（fast-forward）

  • 有分叉，三方合并
    和之前将分支指针向前推进所不同的是,Git 将此次三方合并的结果做了一个新的快照并且自动创建一个新的提交指向它，这个提交被称作一次合并提交，它的特别之处在于他有不止一个父提交。

合并冲突

有时候合并操作不会如此顺利。 如果你在两个不同的分支中,对同一个文件的同一个部分进行了不同的修改，git就没法干净的合并它们。

  1. 创建分叉分支b1、b2且都对文件c的第二行做了写入操作

  2. 合并b1、b2，会提示有无法合并的冲突

  3. 停止合并有冲突的分支：git merge --abort

分支删除

当我们将一个分支b1的工作合并到b2上后，b1就变为了一个无用的分支，这时就可以移除它：git branch -d <branch_name>

或者强制丢弃一个分支：git branch -D -d <name>

分支管理

显示所有分支

git branch，*为HEAD指针

list and last commit

git branch -v

是否merge

已合并到当前HEAD的分支：git branch --merged

未合并到当前HEAD的分支：git branch --no-merged

分支工作流程范例

一些使用git分支的工作模式流程

长期分支

因为 Git 使用简单的三方合并，所以可以简单地在一段较长的时间内反复把一个分支合并入另一个分支，这是长期分支的基础。

简单来说，长期分支就是在整个项目开发周期的不同阶段，可以同时拥有多个开放的分支，可以定期地把某些主题分支合并入其他分支中，即：

  • master主分支保留stable稳定版本代码
  • 分支develop/next为稳定性测试版本，稳定后merge到master
  • 分支topic为开发版本，保留功能后merge到develop

也就是分级的流水线流程

长期分支常用于大型/长久型项目，使用长期分支的优点是保持项目master足够稳定，并可以维护不同稳定性层次的版本。

主题分支

主题分支对任何规模的项目都适用，它是一种短期分支，被用来实现单一特性或其相关工作。

即主题分支常用于一个功能使用了x*y个解决途径实现后选择一个路径merge到master。
